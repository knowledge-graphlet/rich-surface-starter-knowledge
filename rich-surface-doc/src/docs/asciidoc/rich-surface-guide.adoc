== About This Document

This guide documents the *RichSurfaceTerms* starter knowledge: the content set
that defines the meanings, purposes, and patterns backing the chronology-backed
rich interaction surface and the Scriptorium
(`IKE-Network/ike-issues#806`–`#812`).

The authoritative artifact is the *ledger* — the Java source in
`rich-surface-terms`, organized in functional sections (`ConceptSet`,
`MeaningAndPurposeSet`, `PatternSet`) composed by the set's
`KnowledgeSetSource` — an append-only record of declared stamps and versioned
declarations. Building the module replays the ledger; the released artifact is
a *change set*. A starter set is simply a change set whose base is the empty
store.

== The Ledger Model

Three disciplines govern the source:

Declared stamps, inline::
The ledger reads time-major: a stamp — an explicit (status, time, author,
module, path) tuple with a literal time — is declared inline at its position
in the ledger, and the edits under it follow, across any components. Stamp
identity derives from the tuple, so restating the same tuple anywhere is the
same stamp, and no shared stamp constants are needed. Retirement is a new
inactive stamp; stamps are appended, never edited.

Birth-FQN identity::
A concept's identity is `T5(setUuid, fullyQualifiedNameAtBirth)` — the
*knowledge set's* UUID is the type-5 namespace every identity derives from —
so identity cannot be minted without a meaningful name, and one UUID replaces
one literal per concept. The binding constants in `RichSurfaceTerms` are
handles derived from birth FQNs, never independent identity sources. A birth
FQN is never reused within the knowledge set, even after retirement.

Append-only, resumable declarations::
Once released, a declaration is permanent: revisions and retirements append
new `.at(stamp)` scopes; nothing is deleted. A component is pulled back up
by its birth FQN at any later point in the ledger and simply continues —
nothing is restated — so the source composes as one continuous ledger. The
file reads as history; the resolved current state is the draft report's job.

== What This Set Provides

This first release is the *journal chronology model* — the terminology the
chronology-backed surface needs to exist at all (`IKE-Network/ike-issues#807`).
It answers one question: what does it take to record a conversation as
versioned knowledge rather than a transcript? The comprehensive per-concept
reference — identicons, definitions, logical axioms, parents, children, and
identifiers — is the *Referenced Koncepts* glossary at the end of this guide,
generated from the ledger; this section is the orientation to it.

Three groups of content, and why each is here:

* *The element kinds* root at k:JournalElement[] and branch into
  k:ProseElement[], k:ComponentListElement[], k:ReferenceElement[], and
  k:TombstoneElement[]. A block's kind is not a field value — it *is* which
  pattern the block's semantic uses, so these concepts exist to make the kinds
  classifiable and navigable like any terminology. The reference kind is the
  load-bearing one: a k:ReferenceElement[] holds live knowledge by reference,
  never a copy, which is what lets a conversation edit the graph rather than
  talk about it.

* *The field terminology* — the meanings a pattern's fields carry
  (k:JournalElements[], k:ProseContent[], k:ListedComponents[],
  k:ReferencedChronology[]) and the purposes they serve (k:ElementOrder[],
  k:ElementContent[]). Purposes are deliberately shared across patterns: when
  two patterns genuinely order or carry content the same way, they cite the
  same purpose concept, so the set stays queryable by purpose rather than
  accreting near-duplicates.

* *The patterns* wire that terminology to existing datatypes — the set mints no
  datatype of its own. The k:JournalManifestPattern[] holds a conversation's
  ordered elements as a component-id list, so inserting or reordering a block
  writes a new manifest version and replay is walking those versions in time;
  the element patterns each carry one block's content.

How the set is used: a conversation is a journal-anchor concept; its manifest
orders element semantics; each element either embeds its content or references
a live chronology. Later waves add the role kinds, module policies, and path
classes that turn the surface into a working sandbox — those are enumerated,
in delivery order, in the *RichSurfaceTerms Inventory*
(`dev-rich-surface-terms-inventory`, Kompendium Part IV).

== Building and Releasing

`mvn clean verify` replays the ledger into an ephemeral store and verifies the
declared chronologies, including replay idempotence. The change-set export
module (planned) attaches the `.pb.zip` artifact — full, and delta against the
prior release — with the release verifier enforcing the append-only guards:
presence of every released declaration, no birth-FQN reuse, and no mutation of
released history.
